# Lessons

- **Cuphead has a native macOS build.** Don't assume the client needs Whisky/Wine; default the Mac installer to the native Steam path `~/Library/Application Support/Steam/steamapps/common/Cuphead/` and use BepInEx_macos_universal, not BepInEx_win_x64.
- **Don't refer to the second player as "the friend".** Use "the client", "the partner", "the second player", or just "the other PC". Neutral language only.
- **Overlay vs console.** The IMGUI overlay shows mode/Rewired ids/sequence/buttons/axes; the BepInEx console shows log lines like "CoopClient: handshake ok". Don't conflate them in test instructions.
- **BepInEx 5.x preloader hits libc.so.6 on macOS — Mono dllmap is the real fix, not a version downgrade.** Symptom: `preloader_TIMESTAMP.log` appears in `Cuphead.app/Contents/MacOS/` with `DllNotFoundException: libc.so.6`, and no `BepInEx/LogOutput.log` ever gets created. Root cause: BepInEx's `PlatformUtils.SetPlatform()` first reads `Environment.OSVersion.Platform.ToString()`, which on Cuphead's Mono returns `"Unix"`. The string contains "unix", so BepInEx classifies the host as Linux and calls `uname_linux` — declared as `[DllImport("libc.so.6")]`. macOS has no `libc.so.6`, so the import fails before any plugin loads. Both 5.4.22 and 5.4.23.5 have this — it's not a version-specific bug. Fix: add `<dllmap dll="libc.so.6" target="/usr/lib/libSystem.dylib" os="osx"/>` to BOTH `Cuphead.app/Contents/Mono/etc/mono/config` AND `Cuphead.app/Contents/MonoBleedingEdge/etc/mono/config`. (The stock Mono dllmap only redirects the bare `libc` alias, not the full `libc.so.6` name — different strings as far as Mono's import resolver is concerned.) `setup-mac.sh` does this automatically. Also: 5.4.22's `run_bepinex.sh` ships with `executable_name=""`; on macOS this must be set to `Cuphead.app`. (5.4.23.5 auto-detected; 5.4.22 doesn't.)
- **Don't expect a Terminal window to pop on macOS.** BepInEx Windows builds spawn a separate console; the macOS unix builds log to `BepInEx/LogOutput.log` only unless `Logging.Console.Enabled = true` is set in `BepInEx/config/BepInEx.cfg` AND the launching shell has a TTY (Steam doesn't). The right "did it load?" signal on Mac is the existence and contents of `LogOutput.log`, not a popup window.
- **F10/F11 are unreliable hotkeys on macOS, even with Fn.** macOS intercepts them for Mission Control / Show Desktop / etc. before Unity's input system sees them, and toggling System Settings → Keyboard → "Use F1/F2 etc as standard function keys" is not enough on most setups. Use letter keys for any plugin hotkey on Mac (`J` for Connect, `K` for Disconnect in this repo). F9 happens to work in practice but isn't guaranteed across macOS versions. ModConfig defaults stay F9/F10/F11 because Windows doesn't have this issue; setup-mac.sh writes letter-key overrides into the pre-seeded cfg.
- **LiteNetLib's NatPunchModule needs System.Runtime.Serialization on Mono.** `NetManager`'s constructor eagerly news up `NatPunchModule`, which calls `NetPacketProcessor.SubscribeReusable<NatIntroduceResponsePacket>()` → NetSerializer's reflection codepath → `FormatterServices.GetUninitializedObject` (lives in `System.Runtime.Serialization`). Cuphead's Unity 2017 Mono distribution doesn't ship that assembly, so `NetManager`'s constructor throws `FileNotFoundException`. Symptom: `CoopHost.Start` / `CoopClient.Start` never reach `Started UDP …` logging; mock client gets `disconnected: ConnectionFailed` because nothing is listening. Fix: Harmony-prefix `LiteNetLib.NatPunchModule.ctor(NetSocket)` to skip its body (`return false`). NatPunch isn't needed when running over ZeroTier or LAN, so this is purely a workaround for the Mono assembly gap. See `src/Coop/LiteNetLibPatches.cs`. Don't bump LiteNetLib past 0.9.5.2 without re-checking — the eager NatPunch init is a 0.9.x quirk; 1.x lazy-inits.
- **Application.runInBackground defaults to false in Cuphead** — so `Update`/`LateUpdate` pause when the window loses focus. With that, alt-tabbing from Cuphead to a terminal freezes the host's snapshot pump and the network keepalive's timeout eventually disconnects the client. Set `Application.runInBackground = true` in Plugin.Awake to keep the simulation, network polling, and snapshot tick running while the game is in the background. LiteNetLib's UDP socket reception isn't affected — only the LiteNetLib `PollEvents` call (driven from Update) is, so the issue presents as "snapshots stop, connection eventually times out".
- **`PlayerManager.DoesPlayerExist` is the right gate for transform sync, not `PlayerInput.Init`.** `PlayerInput.Init` runs as soon as Cuphead binds the Rewired.Player handle to a PlayerId — happens during the player-select/join screen before a level actually starts. But `PlayerManager.GetPlayer(id)` returns a fully-constructed `AbstractPlayerController` only once the player controller is spawned in a level. Sampling transforms before that returns `DoesPlayerExist=false`. Add diagnostic logging that exposes which check is failing (`DoesPlayerExist=false` vs `transform=null` vs exception) so testers can tell apart "wrong scene", "mid-spawn race", and "API broke".
- **No single boss base class in Cuphead.** Each level's boss is its own type (`RumRunnersLevelMobBoss`, `ChessBOldBLevelBoss`, …). Don't try to enumerate "bosses" — instead enumerate `AbstractLevelEntity` (the shared base for level gameplay objects) and filter to those with descendant `Animator` components. That catches bosses, mini-bosses, animated set pieces, and avoids passive scenery / coins / parry pickups.
- **Identify entities cross-machine by FNV1a32 of scene-relative hierarchy path.** Unity instance IDs differ per process. `string.GetHashCode` differs across .NET versions. FNV1a32 is deterministic and stable, gives 32-bit IDs that fit cleanly in network packets. Both sides walk their own scene independently and the hashes line up because the input strings are identical scene-loaded paths.
- **Refresh entity caches periodically, not just on `sceneLoaded`.** Boss phase transitions (HildaBerg → Moon → UFO) instantiate new GameObjects without firing `sceneLoaded`. A 2-second tick from `Plugin.LateUpdate` re-walks `FindObjectsOfType<AbstractLevelEntity>` and refreshes the path-hash → ref dictionary. Cost is ~25ms × 0.5/s (~1-2% CPU). Dedup the "cached N entities" log line so periodic refreshes in a stable scene don't spam the in-game overlay tail.
- **`Animator.Play` only on state change or significant phase drift.** Calling `Play(hash, 0, time)` every frame resets `normalizedTime` to the supplied value, producing a stuttering animation. Only call when `current.fullPathHash != hash` OR when `Mathf.Abs(currentNormTime - hostNormTime) > 0.15`. Same logic for player and entity sync paths.
- **Setting `Mode = CoopMode.Client` is enough to activate client-side patches without an actual host.** Useful for verifying M9 (damage suppression), pause-input suppression, etc. on a single PC: press F10 to dial a non-existent host. Connection times out / sits dialing forever, but `CoopState.Mode = Client` was set the moment `Connect` ran, so prefix patches gated on `Mode == Client` are live. Doesn't verify M5/M6/M8 client-side behavior (those need a real host streaming snapshots).
- **`PlayerDamageReceiver.OnDeath()` is just `state = State.Other` — not the actual death sequence.** Cuphead's death animation, level-failure, retry flow lives elsewhere (subscribers to `PlayerStatsManager.OnPlayerDeathEvent`, coroutines on `AbstractPlayerController`, level-specific scripts). Setting HP to 0 via the property setter MAY trigger the chain via `OnHealthChangedEvent` subscribers, but the relationship is implicit. Mirroring death cleanly requires either firing all the right events in order or finding a single `Die()`-like entry point — neither is obvious from a quick decompile. Defer to future M8.5 with proper investigation.
- **`PauseManager.Pause` directly is the right way to freeze the client without showing a pause menu.** It sets `CupheadTime.GlobalSpeed = 0`, calls `OnPause` on registered `AbstractPausableComponent` children, and pauses audio — without bringing up `AbstractPauseGUI`. Calling it from a Harmony patch is fine; gating local Pause/Unpause invocations on a `RemoteDriven` flag lets us tell apart "host says pause" (allow) from "client pressed pause locally" (block).
- **`[DefaultExecutionOrder]` affects ALL methods on the MonoBehaviour, not just `Update`.** v0.7.6 set `[DefaultExecutionOrder(-32000)]` on Plugin so Update would run before PlayerMotor reads input — but that also dragged LateUpdate to run before Cuphead's animator writers, which would then overwrite our forced animator state. Symptom: 2-frame animation flicker even with parameter setters blocked. Fix: split into two MonoBehaviours, the early-order one for Update only (network pump), a separate `[DefaultExecutionOrder(+32000)]` one for LateUpdate (transform + animator writes win the LateUpdate race).
- **Cuphead's player has multiple Animators** in a hierarchy (body, weapon, fx). Only registering the first one found via `GetComponentInChildren<Animator>()` for parameter-block leaves secondary animators free to drift via their own param transitions, producing visible flicker on the cup's outfit / weapon. Register all `GetComponentsInChildren<Animator>()` instead.
- **`AbstractPlayerController` and `MapPlayerController` are different hierarchies.** Level players inherit from the former (with `PlayerStatsManager`, `PlayerDamageReceiver`, etc); world-map players inherit from the latter (no stats / damage system). `PlayerManager.GetPlayer` returns null on the map. Need a fallback `FindObjectsOfType<MapPlayerController>` lookup keyed by `id` to sync map cup positions. (Implemented in v0.7.5.)
- **Cuphead has no single base class for projectiles.** Per-level projectiles like `BaronessLevelBaronessProjectile` extend `BasicProjectile` extends `AbstractProjectile` extends `AbstractCollidableObject`. To capture all projectiles in EntitySync, hook `AbstractProjectile`. (`AbstractLevelEntity` is a sibling class — doesn't catch them.)
- **Cuphead's PlayerSlot `joinState` transitions can be force-set via reflection.** The state-machine's natural path is `NotJoining → JoinPromptDisplayed → JoinRequested → Joined`, driven by an `Update` method that watches for any-player-input button-down. To synthesize a join from a network event without local input, reflect into `PlayerManager.playerSlots[1]`, set `canJoin = true` and `joinState = Joined`, set `controllerState = NoController`, set `Multiplayer = true`, and fire `OnPlayerJoinedEvent` via the backing private delegate field. Clean.
- **Path-hash sibling-index trick for runtime clones.** Static scene objects have stable hierarchy paths that can be hashed for cross-machine identification. Runtime clones get names like `Bullet (Clone)` which collide. Substituting `[<sibling-index>]` for the name segment when it contains `(Clone)` produces hashes that align between host and client AS LONG AS spawn order is deterministic (which it is when M5 keeps animator states aligned and bosses fire at the same animation phase on both sides).
- **EntitySync's "alive hash list".** Streaming positions for up to N entities is enough for visible mirroring, but doesn't tell the client which entities the host considers DEAD. A separate `uint[]` list of "all alive hashes this frame" (no positions) is cheap (~1 KB max for 256 hashes) and lets the client `SetActive(false)` on cached entities not in the list — host-killed enemies disappear on client without waiting for the next 0.5s cache-rebuild cycle.
- **`_cacheValid` initialization gotcha.** Static bool field defaults to `false` in C#. EntitySync's `_cacheValid` was guarded by every Capture/Apply method but never set to `true` after a successful `RefreshCache` — so all three gates early-returned and the entire entity sync layer was silently dead from v0.4.0 through v0.8.0. The host log said "cached N entities" but `tx ents` was always 0. Lesson: when adding a `_xxxValid` flag, either set the default to `true` on first init or unconditionally flip it true at the end of any successful population pass. Cheap diagnostic the test missed: a top-level overlay showing `tx ents=0` despite a non-zero cache log line.
- **Cuphead's `ArcadePlayerMotor.HandleInput` runs in `FixedUpdate`, not `Update`.** And FixedUpdate fires BEFORE Update in a Unity frame. So a network pump that only runs in `Plugin.Update` reads packets ONE PHASE LATE — by the time a network packet's button-down edge is visible to `CoopState.CurrentButtons`, FixedUpdate's `actions.GetButtonDown(2)` has already fired and missed it. AdvanceFrame(LateUpdate) then promotes Current→Previous, and the down-edge is gone forever. Symptom: shoot works (uses GetButton, held semantics), jump doesn't (uses GetButtonDown, edge semantics). Fix: also pump the network in `Plugin.FixedUpdate` so packets are visible BEFORE Cuphead's input read in the same frame.
- **Path-hash with sibling-index can't sync runtime clones.** The "use sibling index for `(Clone)` GameObjects" trick assumes spawn order is identical on both sides. It isn't — RNG, animator timing drift between host and client (even with state mirroring), and gated client input mean projectile #3 on host is literally not the same object as projectile #3 on client. Path-hashes never converge for clones; client never sees host's projectiles. Right answer for runtime clones is host-assigned synthetic NetworkIDs, transmitted to client, with client binding to local instances by closest-position match (or instantiating from prefab if no local exists). v0.8.2 hooks `AbstractProjectile.Awake` on host to assign IDs and on client to register unbound candidates; the binding heuristic is "closest-unbound-within-80-units" the first time we see a NetworkID. Static scene actors (`AbstractLevelEntity`) keep using path-hash because their identities ARE stable.
- **Cuphead's animation controllers manually call `animator.Update(deltaTime)` four times per frame.** `ArcadePlayerAnimationController.Update` (and its Level / Map siblings) explicitly tick the animator forward in addition to Unity's automatic update — once with deltaTime, then three times with 0 to flush transitions. This advances time-based transitions on the animator's state machine REGARDLESS of whether any parameters changed. So our v0.7.6 parameter-setter block on registered animators had no effect on time-based progressions: idle states with Exit Time would auto-transition to default, our LateUpdate would force back to host's state, next frame the controller would advance again — visible 2-frame cup flicker between two animation states even when stationary. Fix: Harmony-prefix-block `Update` entirely on the three player animation controllers when `Mode == Client`. Animator state on client comes purely from host's `Animator.Play` calls in ScenePuppetry.ApplyTo; the controller's frame-by-frame advancement is unwanted.
- **Auto-deploy to *both* Cuphead installs when running solo two-instance tests on one PC.** Two separate game directories (`Cuphead/` and `Cuphead-client/`) means a single AfterBuild copy step only updates one — the other window keeps loading whatever stale DLL was last there. Symptom: launcher starts both windows, one shows the new version in the overlay, the other doesn't. Either: extend csproj to copy to both paths, or always launch from the same directory. v0.8.1 added a second `<Target Name="DeployToClientInstall">` conditional on `$(CupheadDir)-client/BepInEx` existing.
- **HKMP (Hollow Knight Multiplayer) is the canonical reference architecture for "add online to a single-player Unity Mono game via BepInEx."** Their key insight: don't keep two parallel sims in sync. Pick ONE machine to simulate each entity, and have the OTHER machine STOP simulating that entity entirely and just render snapshots. Scene-host pattern: first client to enter a scene becomes authoritative for that scene's enemies; non-scene-host clients don't run enemy AI at all. RNG divergence is sidestepped by design. The Cuphead online mod that does exist (Germanized/CupHeads) hits the same bugs we did because they kept client-runs-local-sim. Don't rebuild that — go HKMP-style for enemies.
- **Input mirroring beats transform-only mirroring for Unity Mono game co-op mods.** Symptom that motivated the v0.8.3 rewrite: client's local P1 input was zeroed by RewiredFocusGate, so client's local sim never fired weapons (no AbstractProjectile.Awake fired → ProjectileSync had no unbound candidates → host's NetworkID stream had nothing to bind to → projectiles invisible on client). Run-and-gun trigger zones never activated on client because client's local P1 stayed at scene origin. Fix: stream host's per-frame Rewired Buttons + Axes for both players in PlayerSnapshot; client's RewiredFocusGate substitutes them into local Rewired reads via the same player-id dispatch host uses for its P2 postfix. Both sides then run the full simulation with the same inputs. RNG drift remains but is correctable via existing position/animator streams. This is the closest you can get to HKMP without a "client doesn't run sim" rewrite — and it's a strict prerequisite for projectile sync to work because client needs LOCAL projectile instances to bind host's NetworkIDs against.
- **Don't reject "decompile + modify" without considering BepInEx preloader patchers.** When evaluating "should we modify Assembly-CSharp directly?" the right answer is "stick with Harmony for ~95% + use BepInEx preloader patchers (Mono.Cecil, runs pre-Unity-load) for the rare 5% that need new fields / changed inheritance / new types." Hand-edited DLLs break on game updates and conflict with other Harmony-based mods (DebugMod, ArchipelagoMod). Decompile fidelity on Unity 2017 Mono is lossy for coroutines (which Cuphead uses heavily for boss patterns). The preloader patcher API is documented at docs.bepinex.dev/articles/dev_guide/preloader_patchers.html and is what HKMP / DebugMod / etc. use when Harmony alone isn't enough.
- **`MonoBehaviour.enabled = false` is the cheapest way to suppress AI on a per-component basis without disabling rendering.** Unity stops calling `Update` / `FixedUpdate` / `LateUpdate` / coroutines on a script when `enabled` is false, but the GameObject stays active so its `Animator`, `SpriteRenderer`, `Transform`, etc. keep working — host's streamed transform + animator-state-Play continues to render the entity correctly. v0.9.0 uses this for `AbstractLevelEntity`-derived AI scripts on client: client stops running boss/enemy AI in parallel with host, eliminating the RNG-driven attack-pattern divergence that no amount of position/animator stream correction could fix. Track which components you disable in a `Dictionary<int, MonoBehaviour>` keyed by `GetInstanceID()` so you can re-enable on disconnect — otherwise next single-player session has dead bosses.
- **Deterministic type-hash via `FNV1a32(Type.FullName)` removes the need for a "type registry" sync packet.** When host streams "spawn entity X" to client, client needs to know what prefab to Instantiate. Naive approach: send a TypeRegistry packet mapping local TypeId → Type.FullName. Cleaner: skip the packet, just hash `Type.FullName` to a uint on both sides. As long as host and client run the same Cuphead build, both compute the same hash for the same type. Client builds its own `Dictionary<uint, GameObject>` registry by walking `FindObjectsOfType<AbstractLevelEntity>()` at scene load and hashing each instance's `GetType().FullName`. When host streams a TypeId client doesn't have a template for, registry miss is the failure mode (skip render). When client has it, `Object.Instantiate(template)` produces a working clone. v0.9.0's `TypeRegistry.cs`. Bonus: stateless across reconnects — no handshake state to lose.
- **BepInEx's disk logger drops `LogDebug` by default.** `[Logging.Disk] LogLevels` defaults to `Fatal, Error, Warning, Message, Info` — all our `Verbose`-gated diagnostics use `LogDebug` and silently vanish from `LogOutput.log`. Symptom: handshake lines present, zero `rx state`/`tx input` lines despite Verbose=true, looks exactly like "snapshots not flowing". Set `LogLevels = All` in `BepInEx.cfg` when log-verifying traffic.
- **Cuphead runs headless with `-batchmode -nographics` — full plugin + network stack, no display needed.** Normal windowed launch on a locked/display-off desktop hangs forever pre-window: process alive, ~1s CPU, no MainWindowTitle, `output_log.txt` says "All resolution switches have failed / DX11 could not switch resolution". Wasted two launch cycles at 5 AM before checking the Unity player log. For autonomous transport/handshake tests, batchmode is strictly better: boots in seconds, AutoStart + logs give full verification.
- **Goldberg Steam emu can solo-test the Steam P2P transport on one PC.** Goldberg implements ISteamNetworking P2P between its own instances. Per-install `steam_settings/` beside the dll with `force_steamid.txt` (distinct SteamID64s!), `force_account_name.txt`, `force_listen_port.txt` (distinct ports), and `custom_broadcasts.txt` (loopback `127.0.0.1:<other-port>`) gives two local instances that discover and dial each other exactly like two real Steam users. The `.goldberg` / `.steam_orig` copies of `steam_api64.dll` sit in both installs — swap active dll to choose. Remember to restore `.steam_orig` before real two-PC Steam play.
- **Steam P2P drops unreliable messages over ~1200 bytes — size-gate the delivery mode.** `k_EP2PSendUnreliableNoDelay` silently caps at 1200B; entity-heavy StateSnapshots can exceed that. SteamTransport sends Sequenced payloads >1150B as `k_EP2PSendReliable` (Steam fragments reliable up to 1MB) while keeping the DATA_SEQ latest-wins header so stale ones are still dropped on receive. Also: Steam P2P is connectionless — HELLO/BYE/PING/PONG opcodes synthesize the session, and the game's own `SteamManager` (internal class — reflection to read `Initialized`) already pumps `SteamAPI.RunCallbacks()` so `Callback<P2PSessionRequest_t>` dispatches for free.
- **`Object.Instantiate(liveSceneInstance)` works as a poor-man's prefab spawner.** Unity's prefab references are normally serialized fields you'd set in the editor. Modders don't have those references — but the live entity in the scene is itself instantiable. Cloning a live AbstractLevelEntity / AbstractProjectile produces a new GameObject with all the same components, properly initialized. Watch out: scene-bound singleton state (e.g., LevelManager refs) might break on a clone, but Cuphead's gameplay objects mostly don't have that pattern. v0.9.0 EntitySync's spawn-from-host path works this way. If a specific entity type breaks on Instantiate, fall back to the existing path-hash-miss skip behavior — visible as a missing entity but not a crash.
